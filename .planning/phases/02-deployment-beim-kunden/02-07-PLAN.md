    # RUNBOOK - Vor-Ort-Setup beim Kunden

    **Modus:** Vor-Ort-Termin bei der Tankstelle (D-01 Primaer-Modus).
    **Ziel-Dauer:** 30-45 Minuten.
    **Vorbereitung:** siehe Preambel unten.

    ---

    ## ⚠ BLOCKER vor Termin-Start

    Der Vor-Ort-Termin darf **NICHT starten**, solange folgende Punkte nicht freigegeben sind:

    - [ ] **AVV/DSGVO-Freigabe:** siehe `AVV-CHECKLIST.md` - alle 5 Abschnitte muessen ✅ sein, Freigabe-Signatur (Betreiber + Vizionists) vorhanden. Ohne AVV/DPA-Rahmen keine Live-Verarbeitung.
    - [ ] **Pre-Deployment-Test:** `PRE-DEPLOYMENT-TEST-REPORT.md` ist vollstaendig ausgefuellt und "Bereit fuer Vor-Ort-Termin?" mit "Ja" beantwortet.
    - [ ] **USB-Stick oder scp-Bereitstellung:** Deployment-Paket `dist/deployment-paket-v1.0.0/` liegt physisch auf USB oder ist auf Kundenserver per scp uebertragbar. SHA256 der Tarball-Datei ist notiert.
    - [ ] **Kunden-Interview (optional vorab):** `KUNDEN-INTERVIEW.md` durchgefuehrt und Antworten sind in `deployment/context.md.tankstelle-erstversion.md` eingeflossen. Wenn nicht vorab: das Interview passiert im Schritt 4 des Termins (aber Termin verlaengert sich um ~20-30 Min).
    - [ ] **Anthropic-API-Key** mit Guthaben ist bereit. Vizionists bringt den Key mit (im Namen des Betreibers erstellt).

    ---

    ## Vorbereitung (5 Min vor Termin)

    Auf Vizionists-Laptop:

    ```bash
    # Falls das Deployment-Paket noch nicht existiert:
    cd /path/to/kiemailagent
    bash scripts/build-deployment-package.sh v1.0.0
    # Erwartung: dist/deployment-paket-v1.0.0/ existiert.

    # USB-Stick vorbereiten:
    cp -r dist/deployment-paket-v1.0.0/ /media/usb-stick/
    umount /media/usb-stick
    # SHA256 notieren:
    cat dist/deployment-paket-v1.0.0/kea-tankstelle-v1.0.0.tar.sha256
    ```

    Alternativ (scp-Modus):
    ```bash
    scp -r dist/deployment-paket-v1.0.0/ user@kunden-server:/tmp/
    ```

    ---

    ## Schritt 1: Server-Check (5 Min)

    Am Kundenserver, mit User-Session (nicht root):

    Folge `PREFLIGHT.md` durch die 7 Sektionen (OS/Docker/RAM/Disk/IMAP/Anthropic/.env). Alles muss 🟢 oder 🟡 mit dokumentierter Notiz sein.

    **Rollback:** kein Rollback noetig - Preflight aendert nichts.

    **Zeit-Signal:** Nach 5 Min sollte der Check durch sein. Wenn Docker fehlt und installiert werden muss: +10 Min = 15 Min gesamt. Das ist noch im Budget.

    ---

    ## Schritt 2: USB-Transfer + Docker-Image laden (3 Min)

    ```bash
    # USB einstecken, mounten:
    sudo mkdir -p /mnt/usb
    sudo mount /dev/sdX1 /mnt/usb   # X ersetzen (z.B. sdb1 - `lsblk` zeigt Geraet)

    # Deployment-Paket kopieren:
    sudo mkdir -p /opt/kea
    sudo cp -r /mnt/usb/deployment-paket-v1.0.0/. /opt/kea/
    sudo chown -R $USER:$USER /opt/kea

    cd /opt/kea

    # SHA256-Integritaet pruefen (T-02.04-01 Mitigation):
    sha256sum -c kea-tankstelle-v1.0.0.tar.sha256
    # Erwartung: "kea-tankstelle-v1.0.0.tar: OK"

    # Image laden:
    docker load -i kea-tankstelle-v1.0.0.tar
    docker images | grep kea-tankstelle
    # Erwartung: kea-tankstelle:v1.0.0 wird gelistet

    # USB unmounten:
    cd /
    sudo umount /mnt/usb
    ```

    **Zeit-Estimate:** `docker load` dauert ~10-30 Sek (Tarball ~200 MB). Rest ist Copy-Paste.

    **Rollback:** `rm -rf /opt/kea && docker rmi kea-tankstelle:v1.0.0`. Weniger als 30 Sek.

    ---

    ## Schritt 3: `.env` aus Template + Kunden-Credentials (5 Min)

    ```bash
    cd /opt/kea
    cp deployment/kunde-env.example .env
    chmod 600 .env
    nano .env
    ```

    Im Editor mit dem Betreiber gemeinsam ausfuellen (nur diese 5 Felder pflicht):

    1. `IMAP_USER=` - **Kunden-E-Mail-Adresse** (z.B. `info@tankstelle-mustermann.de`)
    2. `IMAP_PASSWORD=` - **App-Password oder normales Mail-Password** (Betreiber tippt selbst, damit Vizionists es nie sieht)
    3. `IMAP_DRAFTS_FOLDER=` - **Wunsch-Ordner** (Vorschlag: `KI-Entwuerfe` als dedizierter Bot-Ordner, oder Standard `Entwuerfe`/`Drafts` je nach Provider)
    4. `OWN_EMAIL_ADDRESS=` - meist identisch zu `IMAP_USER`
    5. `ANTHROPIC_API_KEY=` - Vizionists tippt (kommt aus mitgebrachtem Notizzettel oder Password-Manager)

    Optional wenn Auto-Detect (D-23) fehlschlaegt (unwahrscheinlich fuer bekannte Provider):
    - `IMAP_HOST=` - IMAP-Server (z.B. `imap.ionos.de`)
    - `IMAP_PORT=993`
    - `IMAP_USE_SSL=true`

    Nach Speichern (Ctrl-O, Enter, Ctrl-X in nano):
    ```bash
    ls -la .env
    # Erwartung: -rw------- (Permissions 600)
    ```

    **Zeit-Estimate:** 3-5 Min inklusive Betreiber-Tippen des Passworts.

    **Rollback:** `rm /opt/kea/.env` + Restart (Container faellt Fail-Fast in RuntimeError, kein Auth-Versuch mit falschen Daten).

    ---

    ## Schritt 4: `context.md` finalisieren (5-15 Min)

    ```bash
    cp deployment/context.md.tankstelle-erstversion.md context.md
    nano context.md
    ```

    Gemeinsam mit Betreiber die 6 Sektionen durchgehen (siehe `KUNDEN-INTERVIEW.md` fuer Fragen-Referenz):

    1. About - passt der Text? Firmenname, Standort, Charakter?
    2. Oeffnungszeiten - stimmen die uebertragenen Werte?
    3. Angebote/Preise - Waschanlage-Preise, Werkstatt-Leistungen ergaenzen wenn im OSINT-Rohbau nicht aufgefuehrt
    4. FAQ - 2-3 typische Fragen mit Antwort-Beispielen aus Betreiber-Erfahrung
    5. Ton - Sie/Du, Formulierungs-Style
    6. Signatur - Name, Firma, Adresse, Telefon, E-Mail

    **Wichtig:** context.md ist ein lebendes Dokument (D-10). Beim ersten Termin muss es nicht perfekt sein - Phase 3 schleift nach.

    **Zeit-Estimate:** 5-15 Min. Wenn `KUNDEN-INTERVIEW.md` vorab durchgefuehrt wurde: eher 5 Min (nur Review). Wenn nicht: 15+ Min.

    **Rollback:** `cp deployment/context.md.tankstelle-erstversion.md context.md` — Vorlage liegt im Deployment-Paket unter `/opt/kea/deployment/`, kein Git noetig (D-05: kein Git am Kundenserver).

    ---

    ## Schritt 5: Container starten (2 Min)

    ```bash
    cd /opt/kea
    docker compose up -d

    # Sofortige Logs pruefen:
    docker compose logs -f agent
    ```

    Erwartete Log-Events innerhalb 5-10 Sekunden:
    - `startup` mit `imap_host=<auto-detected-host>`
    - `imap_connected` mit `user=<IMAP_USER>` - **HIER SICHT- UND WICHTIGSTES SIGNAL: kein `auth_failed`!**
    - `poll_start` mit `since=<isoformat>` und `folder=INBOX`
    - `poll_done processed=0` (Nach Backfill-Fenster, keine neuen Mails noch)

    **Bei `auth_failed`:** IMAP-Password oder Auto-Detected-Host falsch. Rueck zu Schritt 3, `.env` korrigieren (oder manuellen `IMAP_HOST=`-Override eintragen), `docker compose restart agent`.

    **Zeit-Estimate:** 30 Sek Startup + 2 Min Log-Beobachtung.

    **Rollback:** `docker compose down` - Container weg, State-Volume bleibt (kein `-v`).

    ---

    ## Schritt 6: Live-Test - Betreiber-Testmail vom Handy (10 Min)

    Der Betreiber holt sein privates Handy und oeffnet seine private Mail-App (z.B. iOS-Mail, Gmail-App). Er schickt eine realistische Test-Mail an die Tankstellen-E-Mail-Adresse - **NICHT** vom Vizionists-Test-Account (D-11/D-12 - Rationale: realer Absender, keine Fake-Rest im Postfach).

    **Vorschlag Betreff/Body:** "Guten Tag, ab wann haben Sie am Sonntag geoeffnet?" - realistisch, kurz, LLM-Klassifikation sollte klar REPLY_NEEDED liefern.

    Danach:
    - **max. 5 Min warten** (1 Poll-Zyklus).
    - Betreiber oeffnet sein normales Mail-Programm (Web-Interface des Providers, oder Thunderbird/Outlook/iOS-Mail) auf DEM Tankstellen-Account.
    - Betreiber navigiert zum Ordner, der in `IMAP_DRAFTS_FOLDER` konfiguriert wurde (z.B. `KI-Entwuerfe`).
    - Ist da ein Draft mit Bezug zur Oeffnungszeiten-Frage?
      - Wenn ja: 🎉 **Live-Verify erfolgreich.** Draft-Threading auf Original-Mail? Signatur enthalten? Ton passt?
      - Wenn nein: `docker compose logs -f agent` pruefen. Wurde die Mail gesehen (`poll_done processed=1`)? Wurde als REPLY_NEEDED klassifiziert? Wurde APPEND versucht (`draft_appended` oder `drafts_folder_created`)?

    **Auto-CREATE-Verify (D-25):** Bei erster Draft-Erstellung sollte `docker compose logs` das Event `drafts_folder_created` zeigen (weil `IMAP_DRAFTS_FOLDER` neu ist). Wenn nicht sichtbar: Ordner existierte bereits, ist auch OK.

    **Zeit-Estimate:** 10 Min inkl. Wartezeit auf Poll-Zyklus.

    **Rollback:** Falls Test-Mail-Draft nicht entsteht: pruefen ob's an Klassifikation liegt (Prompt-Iteration - `nano prompts/classify.txt && docker compose restart agent`) oder an APPEND-Rechten (dann Kunden-Admin/Provider kontaktieren).

    ---

    ## Schritt 7: Reboot-Test (5 Min)

    ```bash
    # Betreiber ausserhalb der SSH-Session einloggen oder physisch am Server:
    sudo reboot
    ```

    Nach dem Reboot (~1-2 Min Wartezeit):
    ```bash
    # Wieder einloggen, dann:
    docker ps | grep kea-agent
    # Erwartung: STATUS = "Up X seconds/minutes" (nicht "Exited")

    cd /opt/kea
    docker compose logs --tail=20 agent
    # Erwartung: startup + imap_connected + poll_start, alles ohne Fehler
    ```

    Damit ist bewiesen dass `restart: unless-stopped` + `systemctl is-enabled docker` funktioniert (DEP-05 abgehakt).

    **Zeit-Estimate:** 3-5 Min inkl. Reboot-Wartezeit.

    **Rollback:** Falls Container nach Reboot NICHT `Up`: `sudo systemctl enable docker && sudo systemctl start docker`, dann `docker compose up -d`.

    ---

    ## Uebergabe an Betreiber (5 Min)

    Kurze Vor-Ort-Uebergabe:
    - Zeigen wo im Mail-Programm der `IMAP_DRAFTS_FOLDER` ist.
    - Wie Drafts geoeffnet, ggf. editiert, gesendet werden (Standard-Mail-Programm-UX, kein Vizionists-spezifisches Verhalten).
    - Wichtige Hinweise:
      - Der Bot sendet NIE selbstaendig - keine Sorge.
      - Bei Problemen (kein Draft, seltsame Antworten): Vizionists per E-Mail oder Telefon kontaktieren.
      - Nach ~1 Woche macht Vizionists eine kurze Nachschulung (Phase 3), um `context.md` und die Prompts nachzuschleifen.
      - Updates: Vizionists liefert bei Code-Aenderungen einen neuen Tarball per scp oder USB — kein automatisches Update (D-17).
    - Kurzanleitung (aus Plan 03 vorgesehen) folgt spaeter per Mail als PDF.

    Runbook-Abschluss:
    - [ ] Alle 7 Schritte erfolgreich
    - [ ] Betreiber weiss wo die Drafts landen und wie er sie prueft
    - [ ] Vizionists-Kontakt fuer Support ist bekannt (E-Mail + Telefon)

    ---

    ## Kern-Rollback (weniger als 5 Min)

    Wenn IRGENDWAS ausser Auth-Problemen und Prompt-Qualitaet schief geht:

    ```bash
    cd /opt/kea
    docker compose down -v         # Container + Volumes weg
    cd /
    sudo rm -rf /opt/kea           # Alles weg
    docker rmi kea-tankstelle:v1.0.0
    ```

    Zeit: ~2-3 Min. Der Server ist im exakten Zustand wie vor dem Termin.

    ## Alternative fuer Remote-Setups (D-02)

    Fuer zukuenftige Kunden bei denen kein Vor-Ort-Termin moeglich ist:

    1. **Delivery:** statt USB-Stick per **scp** aus Vizionists-Laptop nach Kundenserver:
       ```bash
       scp -r dist/deployment-paket-v1.0.0/ user@kunden-server:/tmp/
       ```
    2. **Video-Call (Zoom/Teams/Meet):** Betreiber und Vizionists sind parallel dran. Betreiber shared den Screen der SSH-Session, Vizionists gibt Kommandos an.
    3. **Kommandos:** Alle 7 Schritte funktionieren wie bei Vor-Ort, ausser:
       - Schritt 2 USB-Transfer entfaellt, stattdessen scp-Vor-Uebertragung.
       - Schritt 3 `.env`: Betreiber tippt Password selbst (kein Screen-Share!), Vizionists gibt die anderen 4 Felder an.
       - Schritt 6 Live-Test: identisch (Betreiber-Handy).
       - Schritt 7 Reboot: Vizionists geht kurz auf `disconnected`, Betreiber macht `sudo reboot`, nach ~2 Min wieder SSH-Login, gemeinsam den Log-Check.
    4. **Zeit-Aufschlag:** ~5-10 Min extra, weil Video-Call langsamer ist als physisch-vor-Ort.

    Kein separates Runbook noetig - dieses Dokument deckt beide Modi ab.
