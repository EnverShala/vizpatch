<objective>
Erweitert den Docker-SDK-Wrapper der WebUI von einem statischen Einzel-Container auf N
dynamisch erzeugte Agent-Container (`vizpatch-agent-<agent-id>`) — inklusive Self-Inspection
der eigenen Mounts (um Host-Pfade/Volume-Namen an neue Container weiterzureichen), Labels
für Zuordnung/Aufräumen, `restart_policy: unless-stopped` (Reboot-Überleben ohne systemd)
und Umbau des Update-Flows auf einen SDK-Loop. Entfernt den statischen `agent`-Service aus
der Compose-Datei (Breaking Change) und ergänzt einen expliziten Volume-Namen.

Purpose: MA-03 (ein Container pro Agent via SDK), MA-04 (getrennter State pro Agent über
Env-Var-Routing). Grundlage für Migration (05.04) und Multi-Agent-Routing (05.05).
Output: Multi-Container-fähiges `docker_ctrl.py`, umgebaute `docker-compose.yml`, Mock-Tests.
</objective>

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Tampering | agent_id als Container-/Volume-/Pfad-Komponente | mitigate | Slug-Whitelist `^[a-z0-9-]{1,64}$` serverseitig vor jeder Namens-/Pfad-Konstruktion; ValueError bei Verstoß (Test-abgedeckt) |
| T-05-05 | Elevation of Privilege | Docker-Socket-Mount der WebUI | accept | Root-Äquivalenz ist Phase-4-Design; unverändert; Basic-Auth-Warnbanner + Doku (nur vertrauenswürdiges LAN) bleiben |

| T-05-07 | Tampering | Falscher Volume-Name durch Compose-Namespacing | mitigate | explizites `name: agent-data` + Self-Inspection statt Hardcoding (Pitfall 1) |
