<objective>
Verdrahtet die Multi-Agent-Fähigkeit in der WebUI: alle Konfig-/Status-/Steuer-Routen bekommen
einen `agent_id`-Parameter, es gibt neue CRUD-Routen (`POST /agents`, Rename, Delete mit
Zwei-Stufen-Bestätigung), das Formular erhält ein Agent-Dropdown (leer = Anlege-Modus, D-50) und
ein LLM-Provider-Fieldset (Anthropic|OpenAI|Google + generisches `LLM_API_KEY` + provider-abhängiger
AVV-Hinweis), und die Status-Kachel wird zur Liste pro Agent. `state_reader.py` wird per-Agent
parametrisiert. Dies ist die sichtbare Ende-zu-Ende-Scheibe: nach diesem Plan kann der Betreiber
im Browser mehrere Agenten mit unterschiedlichen Providern anlegen, starten und getrennt überwachen.

Purpose: MA-02 (Dropdown/CRUD), MA-04 (Status/Buttons pro Agent), LLM-01 (Provider-Dropdown-UI +
LLM_API_KEY-Feld), LLM-04 (AVV-Hinweistext).
Output: umgebaute `main.py`/`state_reader.py`, erweiterte Templates, aktualisierte Endpoint-Tests.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-17 | Tampering | agent_id in Pfad-/Container-Routen | mitigate | Slug-Guard (agents_io._agent_dir / docker_ctrl-Validierung) am Routen-Eingang, ValueError → HTTP 400 |
| T-05-18 | Elevation of Privilege | ungeschützte /agents-Routen | mitigate | Alle neuen Routen mit `Depends(auth.require_auth)`, exakt wie Bestandsrouten |
| T-05-19 | Destructive Action | versehentliches Agent-Löschen | mitigate | Zwei-Stufen-Bestätigung (confirmation == "LÖSCHEN") wie Zero-Reset (D-50) |
| T-05-20 | Information Disclosure | Secrets im HTML nach Save | mitigate | read_env_masked liefert "****"; Provider-Feld ist Passwort-Input, kein Klartext im Rendering |
| T-05-21 | Misconfiguration | Provider gewechselt, Key nicht | transfer | Nicht-blockierende UI-Warnung im Save (Pitfall 7) |
</threat_model>
